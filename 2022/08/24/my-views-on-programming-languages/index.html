My Views on Programming Languages

Posted on 2022-08-24 In Programming

There are many debates about the good and bad aspects of programming languages, and I will present my own viewpoints in this blog. I will also explain the keys to learning a programming language, and my understanding of design patterns.

What are we doing when we program?

When we program, we are expressing ideas. We express them in a programming language to produce code, which can then be turned into a program. People can read this code to understand the ideas.

Let's consider an example. Imagine you wish to create a function to sort data. This might remind you of arranging cards while playing poker. Each time you draw a card, you place it in a particular order. This action ensures the cards in your hand are sorted. Programming is about expressing this idea in a programming language, resulting in code that can sort data.

In another instance, let's say you aim to develop a service. You have certain ideas about the API protocol, data structure, processing logic, database operations, and more. Programming is the act of expressing these ideas in a programming language, thereby creating code for your service based on these ideas.

What Defines a Good Programming Language?

Since we are expressing ideas when we program, a good programming language should allow for simple expression. It should enable me to express my ideas almost as if I'm translating them directly, with the only differences being in word usage and syntax. There should be as few structural alterations or information loss as possible. In this way, it will be simple to get the code from the idea when writing the code, or to restore the coder's idea from the code when reading the code.

In these languages, expressing "User information" is quite simple. These languages accurately mirror my intent, with the syntax being the only varying factor. The overall structure remains unchanged; no information is lost.

However, with certain other programming languages, I either have to give up this intuitive approach or use design patterns. For instance, in some languages, to construct an object, I must write new UserInfo("Jason5Lee", "Jason Lee", userBirthday). This requires me to specify values in a predetermined order of fields, which doesn't mirror my intent, where I identify each value by its field. Reading the code, it's not immediately clear which fields correspond to these three values, which makes it less intuitive.

I could use the Builder Pattern by defining a new builder class and using it to specify values by fields when constructing the object, but this introduces an additional class that wasn't part of my original intent.

The simplicity in expressing ideas defines a good programming language.

The Importance of Programming Languages

Programming languages are more than just tools that allow us to execute commands on a computer. They possess a unique attribute of precision, eliminating ambiguity and enabling concise expression of ideas. Much like how mathematicians use specialized language and notation to accurately depict formulas, programmers utilize programming languages to articulate their thoughts and concepts.

Pseudocode is another great example. Even in cases where there is no need for the computer to execute, describing in code form is easier to read and understand. Pseudocode highlights the precision and conciseness that coding language has over natural language.

These advantages of the programming language make it very important, not something that can be made up for with tools like comments or documentation.

When using a statically typed languages or a static checking tools like Pyright, there's an added advantage. They provide the ability to check for potential errors in our intentions. When you attempt to create a UserInfo with an integer as the username, you’ll get a type error. The error is not just about the type, but also your idea. The idea of creating user information with an integer username is incorrect, as the username should be a string. Type checking enables a cycle of thinking, coding, and checking, which requires the code to be intuitive and straightforward.

Language Simplicity versus Expression Simplicity

Simplicity in programming can be viewed from two perspectives: the simplicity of the language itself and the simplicity of expressing using the language. Often, proponents of "simplicity" are referring to the language's simplicity, using it as a rationale to oppose supporting certain features.

However, the simplicity of expression is equally, if not more, important. It allows ideas to be expressed directly, minimizing the gap between thought and code. This aspect is crucial because the simplicity of a language only impacts the initial learning effort. While a language with numerous features might require more effort to learn, you only need to learn them once.

If the language is more expressive, the process of writing and reading each piece of code becomes less burdensome. If we consider writing n amount of code requires an+b effort, language simplicity primarily reduces b, while expression simplicity reduces a. When n is large enough, it's more beneficial for a to be smaller.

Naturally, no programming language exists where all ideas can be directly expressed. A surplus of features, especially those susceptible to misuse, could harm expression simplicity. However, the importance of being straightforward often gets overshadowed by the emphasis on language simplicity. This trend needs to change.

Also, some complexities are objective and can't be ignored. Ignoring them only makes it more complicated as you'll still need to deal with them, but without representation in the code. For instance, consider the optional birthday field in user information. Many languages lack a way to express optionality. Non-existent values like null or nil can be a valid value for many types, even when they're not supposed to be optional. But that doesn't mean the complexity doesn't exist. You'll still need to consider whether a value is optional or not and how to handle it when it doesn't exist. Without a feature to express optionality, handling this becomes even more challenging. Ignorance is not a strength. Misinterpreting ignorance for simplicity can lead to bigger problems.

This is why I recommend using statically typed languages, a static checking tools, beside the points in the previous section. Knowing the type of a value is essential for determining how to handle it. If this isn't made clear, it can result in problems.

This is also why I prefer Rust as a low-level language. When manually managing memory, the lifetime of a value is an objective complexity that you have to consider and ensure no used-after-free or other mistakes are made. Rust’s borrow checker, while it may be more complex to learn, it simplifies the process of expressing the lifetime of a value, understanding the lifetime of a value when reading the code, and ensuring the lifetime of a value when writing the code with the help of the checker. This simplicity brings a significant advantage that outweighs the complexity of learning it.

The inability to express optional data is often referred to as The Billion Dollar Mistake.

How to Learn a Programming Language?

The key to learn a programming language, in my perspective, is to understand how to express ideas using that language. Take the if statement, for instance; it enables us to express different code executions based on varying conditions, mirroring the "if" in our cognitive processes. Similarly, Object-Oriented Programming (OOP) utilizes inheritance to express the "is-a" relationship. When one class is a kind of another, inheritance is used to illustrate this relationship. Combined with method overriding, we can express polymorphism, where a single method can have different implementations in various kinds of the same class. To gain a better understanding of how I use these OOP and Functional Programming (FP) features to express my ideas, you can visit my blog about OOP and FP.

Understanding Design Patterns

In some instances, an idea may not be directly expressible in a programming language. In these cases, we may want to create a specific code pattern, or a design pattern, to articulate the idea. I believe that a design pattern exists due to the absence of a corresponding feature. Many attribute the existence of design patterns to OOP, primarily because the concept was first introduced in a book titled Design Patterns: Elements of Reusable Object-Oriented Software. This has led to the misconception that design patterns are unnecessary in non-object-oriented programming languages. However, the truth is that design patterns were created to express ideas that are challenging to communicate using only OOP features. Even in non-OOP languages, a design pattern may be required if the idea is challenging to express directly. The Builder Pattern, for example, is used in certain languages that lack the ability to create objects by specifying the value of each field, a feature unrelated to OOP.

Consider the Visitor pattern, which expresses that data has multiple variants, each potentially containing different content. The data will be processed differently, accessing different content, based on its variant. The data consists of fixed variants and their content, but the processing remains unpredictable and extendable. To put it simply, the Visitor pattern defines a visitor interface containing the processes for each form. Then, the data provides a polymorphic visit method that accepts a visitor interface parameter. Each variant implements the visit method by invoking the process method for the variant on the visitor interface. This approach not only expresses the data, its variants, and processing but also ensures all variants are covered (since all interface methods must be implemented).

If a programming language doesn't need the Visitor pattern, it must possess a feature that expresses such data. For example, Kotlin and Scala use the sealed keyword, Rust has an enum feature, and TypeScript and Python's type annotation express it through a union of types. Therefore, in these languages, the Visitor pattern is deemed unnecessary. Otherwise, if a language doesn't have such a feature, the Visitor Pattern is necessary.

One might argue that they could simply use an interface to represent the data and process each variant using down-casting, thus eliminating the need for the Visitor pattern or the language feature. However, OOP or Java does not prevent this approach, so if one thinks this is a solution, they shouldn't complain that OOP or Java introduces this design pattern. Moreover, this is a prime example of inherent complexity. There will always be concerns about whether the data has fixed variants, what variants the data encompasses, and whether all variants are addressed in processing. Without clear notation for these aspects, it becomes more challenging to handle them correctly.

When studying design patterns, my advice is to concentrate on what they are trying to express.
